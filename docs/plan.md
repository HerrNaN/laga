# laga - MVP Plan

## Project Goal
Build a mobile-first, offline-first shopping list application. The primary objective of the MVP is a fully client-side shopping list that persists in the browser via IndexedDB, works entirely without network after the first load, and is served by a minimal Go backend.

## Decisions

1. **Go Router:** Standard library `net/http` to minimize dependencies.
2. **Go Module:** `github.com/herrnan/laga`.
3. **Backend Role:** The Go backend serves the compiled frontend (`frontend/dist/`) using `//go:embed` in production. During development, it reverse-proxies to the Vite dev server when `VITE_DEV_PROXY` is set. No CORS is needed because frontend and API share the same origin.
4. **Client-Only Data:** All shopping list data lives in the browser. No database, no backend API, no sync for the MVP.
5. **Styling:** Plain CSS for the MVP to keep dependencies minimal.
6. **TypeScript:** Used throughout the frontend where possible (Vite handles transpilation transparently).
7. **Task Runner:** `mise` manages dev tools (`go`, `node`, `pnpm`) and defines project tasks inline in `mise.toml`.
8. **Node Package Manager:** `pnpm`.

## MVP Scope

### Features
- **Add Item:** Free-text input (e.g., "2 kg potatoes").
- **List Items:** Display items with a checkbox.
- **Check / Uncheck:** Toggle item completion state.
- **Offline Usage:** App loads and functions without any network access after the first visit.

### Defer to v2
- Delete all checked items.
- Recipes and cooking mode.
- Item categorization and custom group sorting.
- Backend persistence and sync.
- Advanced conflict resolution (CRDTs, OT, vector clocks).
- Renaming items or changing amounts after creation.

## Architecture & Stack

| Layer | Tech | Rationale |
|-------|------|-----------|
| **Frontend** | Svelte + TypeScript + Vite | Lightweight, reactive, fast build, type safety. |
| **Offline Load** | Vite PWA Plugin (`vite-plugin-pwa`) | Caches the application shell, allowing the SPA to load without network requests. |
| **Local State** | IndexedDB (`idb` npm package) | Async, structured storage larger than `localStorage`, survives reloads. |
| **Backend** | Go 1.22+ (`net/http`) | Static file server only; zero-dependency routing. |
| **Dev Tools** | `mise` (`mise.jdx.dev`) | Manages `go`, `node`, `pnpm` versions and project tasks. |

## Implementation Roadmap

### Phase 0: Scaffold & Tooling
1. Write `mise.toml` pinning Go, Node, and `pnpm` versions. Define project tasks.
2. Initialize `backend/` as a Go module: `go mod init github.com/herrnan/laga`.
3. Initialize `frontend/` with the official Vite Svelte template using `pnpm`, configured for TypeScript.
4. Install `vite-plugin-pwa` via `pnpm` and configure `vite.config.ts` to generate a Service Worker that precaches the app shell.
5. Write `backend/main.go` with `//go:embed dist` to serve the compiled frontend, and reverse-proxy to Vite dev server when `VITE_DEV_PROXY` is set.
6. Create a placeholder `backend/dist/index.html` so `go build` succeeds before the frontend is compiled.

### Phase 1: Frontend Foundation
1. Define shared types in `frontend/src/types.ts` (e.g., `Item` interface).
2. Create `frontend/src/stores.ts`: a Svelte writable store holding the array of items.
3. Create `frontend/src/db.ts`: wrap IndexedDB (using the `idb` package) to read/write the items array. On app mount, hydrate the store from IndexedDB; any store change writes back to IndexedDB.
4. Build `frontend/src/lib/AddItem.svelte`: an input field and button that parses free text and appends a new item (with client-side UUID) to the store.
5. Build `frontend/src/lib/ItemList.svelte`: renders items from the store with checkboxes. Each toggle updates the store directly.
6. Assemble `App.svelte` with the components and a basic mobile-first layout.

### Phase 2: Polish
1. Implement plain CSS with a mobile-first mindset: large tap targets (`min-height: 48px`), simple flexbox layout, readable font sizes.
2. Ensure `manifest.json` has `display: standalone` and appropriate icons for a mobile-like experience.
3. **Verification:** Disable network, reload the browser, add and check items, reload again. The state must survive.

## Research Notes (for v2)

- **Backend Persistence & Sync:** When adding a backend database, research offline-to-online synchronization strategies:
  - Conflict-free Replicated Data Types (CRDTs).
  - Operational Transformation (OT).
  - Per-item vector clocks or logical timestamps.
  - Event sourcing with replay.
- **Delete Checked:** Implement a "clear completed" action that removes all checked items in one go.
- **Recipes:** Add recipe creation, listing, viewing, and a cooking mode with step tracking.
